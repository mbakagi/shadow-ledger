Monolithic client: `index.html` is the application shell (HTML + Tailwind `<style type="text/tailwindcss">` + inline SVG icons), `app.js` holds all runtime logic in an IIFE, and `firebase-config.js` initializes the compat SDK and exposes global `db`/`auth` references. The app follows a thin MVC pattern inside one file:
- State: a plain `State` object (`items`, `filteredItems`, `currentPage`, `selectedIds`, `locations`, …) kept in memory.
- DAL layer (`DAL` object): abstracts Firestore reads/writes via `onSnapshot` listeners for `inventory` and `locations`, batch writes, and a `transactions` log collection; it also generates IDs and seeds default locations.
- View: DOM refs collected in a `dom` map; rendering functions (`renderTable`, `renderRow`, `renderDashboard`) update only changed nodes to preserve inline-edit focus.
- Controller: event handlers in `bindEvents()` dispatch to CRUD helpers (`saveFieldSilently`, `adjustStock`, `saveItem`, `deleteItem`).

Persistence strategy is dual-path: when authenticated, data lives in Firestore (`inventory`, `locations`, `transactions` collections) with offline persistence enabled via `enablePersistence({ synchronizeTabs: true })`; when unauthenticated the UI is hidden behind a login overlay and no sync runs. A Service Worker (`sw.js`) caches the app shell with cache-first, CDN assets with stale-while-revalidate, and explicitly bypasses Firebase API hosts so auth/Firestore calls always hit the network. Security is enforced server-side by `firestore.rules` which require `ownerId == request.auth.uid` on every document. Deployment is static: `firebase.json` declares hosting rules and rewrites, while `manifest.json` configures the PWA (standalone display, shortcuts for Add Item / Manifest).
